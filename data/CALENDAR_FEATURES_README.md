# `calendar_features_daily.csv` — Data Dictionary & Temporal Design

## Overview

| Property | Value |
|---|---|
| Rows | **7,670** — one per calendar day |
| Columns | **44** |
| Date range | **2010-01-01 → 2030-12-31** |
| Source | `data/calendar-events/calendar-YYYY.json` (21 files) |
| Generated by | `src/preprocessing/prepare_calendar_features.py` |

The dataset is a **full daily backbone** — every single calendar day gets a
row, not just days with events. This is deliberate: a model must learn that
most days are ordinary (79 % of rows) and that events stand out against that
silence.

---

## 1. What a single row represents

Each row is a **snapshot of one calendar day**, encoded along four dimensions:

```
Row = [ WHERE in time ]  +  [ WHAT happened ]  +  [ HOW MUCH recently ]  +  [ HOW FAR to the next ]
         (temporal ID)         (event facts)         (rolling context)          (proximity)
```

### Concrete example — 2010-01-01 (New Year's Day)

| Column | Value | Meaning |
|---|---|---|
| `date` | `2010-01-01` | Unique day identifier |
| `year` / `month` / `day` | `2010` / `1` / `1` | Gregorian position |
| `day_of_week` | `4` | Friday (0 = Mon … 6 = Sun) |
| `day_of_year` | `1` | First day of the year |
| `month_sin` / `month_cos` | `0.5` / `0.866` | January on the unit circle |
| `year_norm` | `0.0` | Start of the 2010–2030 range |
| `event_count` | `1` | One event on this day |
| `events_list` | `English New Year` | Human-readable names |
| `has_religional_festival` | `1` | This event is a religious festival |
| `has_government_holiday` | `0` | Not a government holiday |
| `cal_gregorian_fixed` | `1` | Date is fixed in the Gregorian calendar |
| `events_last_7d` | `0` | No events in the preceding 7 days |
| `days_since_last_event` | `3` | Previous event was 3 days earlier (Dec 29) |
| `days_to_next_event` | `5` | Next event is 5 days away (Jan 6) |
| `is_event_day` | `1` | **Target — binary** |
| `event_type_label` | `2` | **Target — multi-class** (2 = Religious Festival) |

### Concrete example — 2010-01-02 (no event)

| `is_event_day` | `0` | **Target — binary** |
| `event_type_label` | `0` | **Target — multi-class** (0 = No Event) |

## 2. Column groups explained

### 2A · Temporal identity (9 columns)

These describe *where* the day sits in the calendar. They are **ordinal** —
the model can use them as-is for tree-based models.

| Column | Range | Description |
|---|---|---|
| `year` | 2010–2030 | Calendar year |
| `month` | 1–12 | Calendar month |
| `day` | 1–31 | Day of the month |
| `day_of_week` | 0–6 | 0 = Monday … 6 = Sunday |
| `day_of_year` | 1–366 | Ordinal position within the year |
| `week_of_year` | 1–53 | ISO week number |
| `quarter` | 1–4 | Calendar quarter |
| `is_weekend` | 0 / 1 | Saturday or Sunday |
| `is_leap_year` | 0 / 1 | Year has 366 days |

### 2B · Cyclical (sin / cos) encoding (10 columns)

Ordinal features have a boundary problem: month 12 and month 1 are
numerically far apart, yet they are adjacent on the calendar. Cyclical
encoding fixes this by mapping each periodic feature onto the **unit circle**,
so the distance between any two adjacent values is always equal:

$$
\sin\!\left(\frac{2\pi \cdot x}{T}\right),\quad \cos\!\left(\frac{2\pi \cdot x}{T}\right)
$$

| Column pair | Period $T$ | What it encodes |
|---|---|---|
| `month_sin` / `month_cos` | 12 | Month of year — Dec and Jan are neighbours |
| `day_sin` / `day_cos` | 31 | Day of month |
| `dow_sin` / `dow_cos` | 7 | Day of week — Sun and Mon are neighbours |
| `doy_sin` / `doy_cos` | 366 | Day of year — Dec 31 and Jan 1 are neighbours |
| `week_sin` / `week_cos` | 53 | Week of year |

> **Why two values?** A single `sin` is ambiguous (sin(x) = sin(π−x)).
> The `(sin, cos)` pair uniquely identifies any point on the circle.

### 2C · Linear trend (1 column)

| Column | Range | Description |
|---|---|---|
| `year_norm` | 0.0–1.0 | `(year − 2010) / 20` — captures multi-year growth trends |

Year is **not** cyclical, so it gets a linear normalisation, not sin/cos.

### 2D · Event facts — what happened today (6 columns)

These are only non-zero on event days.

**Multi-hot event type flags** — one column per category (a day can carry
multiple types simultaneously):

| Column | Meaning |
|---|---|
| `has_religional_festival` | At least one religious festival today |
| `has_government_holiday` | At least one government holiday today |
| `has_good_to_know` | At least one observance / awareness day today |

**Multi-hot calendar system flags** — which religious/secular calendar
governs the event (determines whether it shifts date year-to-year):

| Column | Meaning |
|---|---|
| `cal_gregorian_fixed` | Fixed date in the Gregorian calendar (e.g. Republic Day) |
| `cal_hindu_lunar` | Follows the Hindu lunisolar calendar — date shifts every year |
| `cal_islamic` | Follows the Islamic (Hijri) calendar — shifts ~11 days/year |
| `cal_parsi` | Follows the Parsi calendar |
| `cal_nanakshahi` | Follows the Nanakshahi (Sikh) calendar |
| `cal_astronomy` | Tied to an astronomical event (solstice, eclipse, …) |
| `cal_day_relative` | Defined relative to another event (e.g. "day before X") |
| `cal_other` | Calendar system not matched above |

> **Why this matters for ML:** A Gregorian-fixed holiday always falls on the
> same month/day, so `month` + `day` alone predict it. A lunar holiday shifts
> each year — the model needs the calendar-system flag to learn that.

### 2E · Rolling event density — recent history (3 columns)

These capture how "busy" the calendar has been *before* the current day.
They are always **lagged by 1 day** (today's events are not counted) so
there is zero information leakage into training.

| Column | Window | Description |
|---|---|---|
| `events_last_7d` | 7 days | Events in the past week |
| `events_last_14d` | 14 days | Events in the past two weeks |
| `events_last_30d` | 30 days | Events in the past month |

Dataset statistics:

| Statistic | `events_last_7d` |
|---|---|
| Mean | 1.47 |
| Median | 1 |
| Max | 6 |

### 2F · Event proximity (2 columns)

How close is the current day to the nearest event?

| Column | Description |
|---|---|
| `days_since_last_event` | Days elapsed since the previous event day (mean ≈ 4.7, max = 26) |
| `days_to_next_event` | Days until the next event day |

These give the model a **sense of position within a gap** between events.

### 2G · Human-readable audit columns (3 columns — not used as features)

| Column | Description |
|---|---|
| `events_list` | Pipe-separated names of all events on the day |
| `types_list` | Pipe-separated type labels |
| `cal_systems_list` | Pipe-separated calendar system labels |
| `primary_event_type` | Highest-priority event type as a string |

### 2H · Target / label columns (3 columns)

| Column | Task | Values |
|---|---|---|
| `is_event_day` | Binary classification | 0 = no event, 1 = event |
| `event_type_label` | Multi-class classification | 0 = No Event, 1 = Good to Know, 2 = Religious Festival, 3 = Government Holiday |
| `event_count` | Regression / count model | Integer ≥ 0 |

Label distribution:

| `event_type_label` | Label | Count | % |
|---|---|---|---|
| 0 | No Event | 6,057 | 79.0 % |
| 1 | Good to Know | 372 | 4.9 % |
| 2 | Religious Festival | 1,055 | 13.8 % |
| 3 | Government Holiday | 186 | 2.4 % |

## 3. How consecutive rows are related (temporal relations)

The rows are **ordered chronologically** — row $i$ is always one day earlier
than row $i+1$. The three mechanisms that make this explicit are:

### 3A · Rolling windows — carrying the past forward

`events_last_7d` on row $i$ equals the sum of `event_count` on
rows $i-7$ through $i-1$. As you move forward one row, the window slides:
the oldest day drops out and yesterday's count enters.

```
Row index →        i-7  i-6  i-5  i-4  i-3  i-2  i-1   i   i+1
event_count →       0    1    0    0    1    0    1   ?    ?
                    └────────────────────────┘
events_last_7d[i] = 0+1+0+0+1+0+1 = 3      (today excluded)
events_last_7d[i+1] drops i-7, adds i
```

This means the model can see **momentum** — a cluster of holidays in the
past week raises all three rolling counters simultaneously.

### 3B · Proximity — measuring the gap

`days_since_last_event` counts up (+1 each row) during a quiet stretch and
resets to 1 the day after an event. `days_to_next_event` counts down (−1
each row) until the next event and then resets.

```
Date        event_count  days_since  days_to_next
2010-01-01      1            3            5       ← event day
2010-01-02      0            1            4       ← gap begins, counting up/down
2010-01-03      0            2            3
2010-01-04      0            3            2
2010-01-05      0            4            1
2010-01-06      1            5            ?       ← next event day
```

Together the two columns tell the model exactly where it stands in the
*inter-event interval*.

### 3C · Cyclical encoding — enforcing calendar periodicity

The sin/cos pairs ensure that the model perceives December and January as
adjacent, and Saturday and Sunday as adjacent to Monday. Without this, a
gradient-based model would see month=12 and month=1 as maximally distant —
the opposite of reality.

```
          month_sin  month_cos     distance to January
January       0.50      0.866         —
February      0.87      0.500        small
...
November     -0.87      0.500        small
December     -0.50      0.866        same as February  ✓
```

## 4. Temporal train / test split rule

```
Train  2010-01-01  ──────────────────────►  2027-12-31   (6,574 days)
Test                                         2028-01-01  ──►  2030-12-31   (1,096 days)
```

> **Critical:** always split chronologically. A random shuffle would allow
> future dates to appear in the training set, inflating accuracy by leaking
> tomorrow's context into yesterday's model.

## 5. Quick-start usage

```python
import pandas as pd
from sklearn.ensemble import RandomForestClassifier

df = pd.read_csv("data/calendar_features_daily.csv")

FEATURE_COLS = [
    # temporal identity
    "year", "month", "day", "day_of_week", "day_of_year",
    "week_of_year", "quarter", "is_weekend", "is_leap_year",
    # cyclical
    "month_sin", "month_cos", "day_sin", "day_cos",
    "dow_sin", "dow_cos", "doy_sin", "doy_cos",
    "week_sin", "week_cos",
    # trend
    "year_norm",
    # event facts
    "has_religional_festival", "has_government_holiday", "has_good_to_know",
    # calendar systems
    "cal_gregorian_fixed", "cal_islamic", "cal_parsi",
    "cal_astronomy", "cal_nanakshahi", "cal_hindu_lunar",
    "cal_day_relative", "cal_other",
    # temporal context
    "events_last_7d", "events_last_14d", "events_last_30d",
    "days_since_last_event", "days_to_next_event",
]

# ── Temporal split ────────────────────────────────────────────────────────────
train = df[df["date"] <= "2027-12-31"]
test  = df[df["date"] >  "2027-12-31"]

X_train, y_train = train[FEATURE_COLS], train["is_event_day"]
X_test,  y_test  = test[FEATURE_COLS],  test["is_event_day"]

model = RandomForestClassifier(n_estimators=200, random_state=42)
model.fit(X_train, y_train)
print(model.score(X_test, y_test))
```
